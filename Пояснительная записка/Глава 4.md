# Глава 4. Тестирование и проверка работоспособности

## 4.1 Конфигурация системы

Разработанное веб-приложение представляет собой клиент-серверную информационную систему, предназначенную для учета комнатных растений, планирования мероприятий по уходу и организации взаимодействия пользователей в рамках клуба любителей растений. Для обеспечения работоспособности приложения требуется программная среда, включающая серверную часть, клиентскую часть, систему управления базами данных и веб-браузер пользователя.

Серверная часть приложения реализована на языке PHP с применением фреймворка Laravel. Она выполняет обработку HTTP-запросов, проверку прав доступа, работу с базой данных, сохранение изображений, формирование ответов API и выполнение бизнес-логики. Клиентская часть реализована на Vue.js и работает в браузере пользователя. Она отвечает за отображение интерфейса, маршрутизацию между страницами, обработку пользовательских действий и передачу данных на сервер.

В качестве системы управления базами данных применяется PostgreSQL. Выбор реляционной базы данных обусловлен структурированным характером информации, хранимой в системе. Данные о пользователях, ролях, комнатах, растениях, настройках ухода, истории выполненных действий, лайках, подписках и жалобах имеют выраженные связи между сущностями. Использование реляционной модели позволяет обеспечить целостность данных с помощью внешних ключей, ограничений уникальности и индексов.

Общая конфигурация развертывания приложения представлена на рисунке 4.1.

[Рисунок 4.1 - Конфигурация системы: браузер пользователя, Vue.js frontend, Laravel API, база данных PostgreSQL, файловое хранилище изображений]

В типовой конфигурации пользователь обращается к приложению через веб-браузер. Интерфейс загружается как одностраничное приложение, после чего необходимые данные запрашиваются у серверного API. Серверная часть принимает запрос, выполняет проверку авторизации, обращается к базе данных и возвращает ответ в формате JSON. При работе с изображениями сервер дополнительно взаимодействует с файловым хранилищем, где размещаются загруженные пользователями аватары и изображения растений.

Для локального запуска проекта требуется установленная среда PHP, менеджер зависимостей Composer, платформа Node.js, пакетный менеджер npm и сервер базы данных PostgreSQL. В процессе разработки приложение может запускаться через встроенный сервер Laravel и dev-сервер Vite. Для эксплуатации в более приближенной к рабочей среде конфигурации серверная часть может быть размещена на веб-сервере Nginx или Apache, а клиентская часть может быть собрана в статические файлы и отдаваться через тот же веб-сервер.

Указанная конфигурация обеспечивает возможность установки, запуска, проверки и дальнейшего сопровождения приложения. При этом клиентская часть не требует установки дополнительного программного обеспечения на компьютере конечного пользователя, так как работа выполняется через браузер. Такое решение соответствует требованиям доступности и снижает сложность внедрения системы.

## 4.2 Обоснование выбора программного обеспечения

Выбор программных средств выполнялся с учетом назначения приложения, требований к надежности, удобству разработки, расширяемости и поддержке современных подходов к созданию веб-систем. Разрабатываемое приложение должно обеспечивать работу с учетными записями, разграничением доступа, связанными сущностями, изображениями, публичной лентой, административными функциями и интерактивным пользовательским интерфейсом. В связи с этим применяемые технологии должны поддерживать как серверную обработку данных, так и удобную организацию клиентского интерфейса.

Фреймворк Laravel выбран для реализации серверной части, поскольку он предоставляет готовые механизмы маршрутизации, работы с базой данных, валидации запросов, аутентификации, авторизации и тестирования. Использование Laravel сокращает объем низкоуровневого кода и позволяет сосредоточиться на предметной логике приложения. В рамках проекта это выражается в наличии отдельных контроллеров API, моделей предметной области, request-классов для проверки данных, ресурсов для форматирования ответов и политик доступа.

Важным преимуществом Laravel является поддержка Eloquent ORM. Данный механизм позволяет описывать связи между сущностями приложения на уровне моделей и выполнять операции с данными в объектно-ориентированной форме. Для рассматриваемого проекта это особенно важно, так как система содержит большое количество связанных объектов: пользователь имеет роль, комнаты и растения; растение имеет изображения, настройки ухода, историю ухода, лайки и советы; жалобы связаны с пользователем и целевым объектом. Использование ORM повышает читаемость кода и упрощает сопровождение связанной структуры данных.

Для аутентификации применяется Laravel Sanctum. Данное решение подходит для API-приложений, в которых клиентская часть взаимодействует с сервером через HTTP-запросы и передает токен доступа. Токен позволяет серверу определять текущего пользователя и проверять его права. В системе это используется для защиты личных растений, профиля, задач, социальных действий и административной панели.

Клиентская часть реализована на Vue.js. Выбор данного фреймворка обусловлен необходимостью построения интерактивного интерфейса, в котором состояние страниц изменяется без полной перезагрузки документа. Vue.js позволяет разделять интерфейс на компоненты, повторно использовать элементы оформления и удобно связывать данные с визуальным представлением. Для приложения учета растений такой подход является целесообразным, так как интерфейс содержит карточки растений, формы, фильтры, списки задач, календарные блоки, профиль пользователя и административные таблицы.

Для маршрутизации в клиентской части применяется Vue Router. Он обеспечивает переход между страницами приложения и позволяет проверять доступ к защищенным разделам. Для управления состоянием используется Pinia. Данное хранилище применяется для авторизации, растений, задач, социальной активности, административных данных и календарного представления. Использование централизованных хранилищ позволяет отделить состояние приложения от отдельных компонентов и уменьшить дублирование логики.

Для сборки клиентской части используется Vite. Он обеспечивает быстрый запуск проекта в режиме разработки и формирование оптимизированной сборки для эксплуатации. Валидация форм на клиентской стороне реализуется с использованием VeeValidate и Zod. Это позволяет проверять корректность вводимых пользователем данных до отправки на сервер и улучшает удобство работы с формами.

Выбор PostgreSQL в качестве базы данных обусловлен надежностью данной СУБД, поддержкой связей, индексов, транзакций и ограничений целостности. В проекте используется несколько типов связей между таблицами, а также уникальные ограничения для email пользователей, лайков и подписок. PostgreSQL обеспечивает устойчивую работу с такими структурами и подходит для дальнейшего роста количества записей.

## 4.3 Аппаратное обеспечение и требования к вычислительным ресурсам

Разработанное приложение не относится к ресурсоемким системам, выполняющим сложные математические вычисления или обработку больших мультимедийных потоков в режиме реального времени. Основная нагрузка связана с обработкой HTTP-запросов, выполнением запросов к базе данных, передачей JSON-ответов и хранением изображений растений. В связи с этим требования к аппаратному обеспечению определяются предполагаемым количеством пользователей, объемом хранимых данных и интенсивностью загрузки изображений.

Для разработки и локального тестирования достаточно персонального компьютера среднего уровня. На таком рабочем месте одновременно запускаются сервер Laravel, dev-сервер Vite, база данных и браузер. Основная нагрузка при локальной разработке приходится на инструменты сборки frontend, сервер базы данных и выполнение тестов. Для комфортной работы рекомендуется использовать многоядерный процессор, не менее 8 ГБ оперативной памяти и накопитель SSD.

Минимальные и рекомендуемые характеристики рабочего места представлены в таблице 4.2.

| Параметр | Минимальная конфигурация | Рекомендуемая конфигурация |
| --- | --- | --- |
| Процессор | 2 ядра, 2.0 ГГц | 4 ядра и выше |
| Оперативная память | 4 ГБ | 8-16 ГБ |
| Накопитель | 5 ГБ свободного места | SSD, 10 ГБ и более |
| Операционная система | Windows, Linux или macOS | Windows 10/11 или Linux |
| Браузер | Современный браузер с поддержкой ES-модулей | Google Chrome, Microsoft Edge, Mozilla Firefox |

Для серверного размещения минимальная конфигурация может быть небольшой, если предполагается ограниченное количество пользователей. Для учебного проекта и демонстрационного применения достаточно виртуального сервера с 1-2 виртуальными процессорными ядрами, 1-2 ГБ оперативной памяти и 10-20 ГБ дискового пространства. При увеличении числа пользователей и объема изображений требования к дисковому пространству будут расти. Наибольшее влияние на размер хранилища оказывают фотографии растений и аватары пользователей.

Оценка требований к серверным ресурсам представлена в таблице 4.3.

| Сценарий эксплуатации | Процессор | Оперативная память | Дисковое пространство | Назначение |
| --- | --- | --- | --- | --- |
| Демонстрационный запуск | 1 vCPU | 1 ГБ | 10 ГБ | Проверка функциональности и защита проекта |
| Небольшая группа пользователей | 2 vCPU | 2 ГБ | 20-40 ГБ | Использование в рамках учебной группы или небольшого сообщества |
| Расширенная эксплуатация | 2-4 vCPU | 4 ГБ и выше | От 50 ГБ | Работа с большим количеством растений и изображений |

При оценке дискового пространства следует учитывать, что каждая запись растения занимает небольшой объем в базе данных, тогда как изображения могут занимать значительно больше места. В проекте предусмотрен сервис обработки изображений, который может выполнять сжатие файла. Это снижает объем хранилища и уменьшает время загрузки изображений пользователем.

Вычислительная нагрузка серверной части в основном формируется операциями выборки данных для ленты, личных растений, задач, календаря и административной панели. Для ускорения таких операций в базе данных предусмотрены индексы по полям пользователя, публичности растений, датам создания, типам ухода, статусам жалоб и другим часто используемым условиям. Применение индексов уменьшает время выполнения запросов и повышает отзывчивость системы.

Таким образом, разработанное приложение может быть развернуто как на локальном рабочем месте, так и на недорогом виртуальном сервере. При этом архитектура допускает дальнейшее увеличение ресурсов без изменения основной логики приложения.

## 4.4 Проверка работоспособности системы

Проверка работоспособности информационной системы проводится для подтверждения того, что разработанное приложение корректно устанавливается, запускается, выполняет основные пользовательские сценарии и обрабатывает ошибочные ситуации. Проверка включает серверную часть, клиентскую часть, базу данных, маршруты API и пользовательский интерфейс.

На первом этапе выполняется подготовка окружения. Устанавливаются зависимости backend и frontend, настраивается файл окружения, указываются параметры подключения к базе данных, генерируется ключ приложения, выполняются миграции и заполнение базы начальными данными. После этого запускаются серверная и клиентская части приложения. Успешным результатом данного этапа считается доступность API и загрузка интерфейса в браузере.

Последовательность проверки установки представлена на рисунке 4.4.

[Рисунок 4.4 - Последовательность проверки установки: установка зависимостей, настройка окружения, миграции, сидеры, запуск серверов, открытие приложения в браузере]

На втором этапе проверяются основные пользовательские сценарии. Пользователь должен иметь возможность зарегистрироваться, войти в систему, создать комнату, добавить растение, задать параметры ухода, загрузить изображение, просмотреть созданное растение, изменить его данные и удалить запись при необходимости. Дополнительно проверяется формирование задач ухода и создание записи в журнале после отметки выполненного действия.

Контрольный пользовательский сценарий может быть описан следующим образом. После авторизации пользователь открывает раздел личных растений, переходит к форме добавления, вводит название растения, указывает комнату и параметры ухода. После сохранения растение появляется в списке. Затем пользователь открывает раздел задач, видит сформированное мероприятие по уходу и отмечает его выполненным. В результате в истории растения появляется запись о выполненном действии.

На третьем этапе проверяется социальная часть приложения. Для публичного растения проверяется отображение в ленте, просмотр карточки, получение количества лайков и списка советов. Авторизованный пользователь должен иметь возможность поставить лайк, оставить совет, подписаться на другого пользователя и отправить жалобу. При этом для неавторизованного пользователя такие действия должны быть недоступны, что подтверждает корректность разграничения доступа.

На четвертом этапе проверяются административные функции. Пользователь с ролью администратора должен иметь доступ к административной панели, списку пользователей, списку жалоб и метрикам трафика. Администратор должен иметь возможность обработать жалобу, изменить ее статус и добавить комментарий. При этом обычный пользователь не должен иметь доступ к административной странице. В случае попытки перехода к административному маршруту он должен быть перенаправлен в доступный раздел.

Для серверной части применяются автоматизированные тесты Laravel. Они позволяют проверить работу API без ручного прохождения всех сценариев через интерфейс. В проекте предусмотрены функциональные тесты для публичного чтения, загрузки медиафайлов, административной и кэширующей инфраструктуры, модерации, а также OpenAPI-контракта. Наличие таких тестов позволяет быстрее обнаруживать ошибки при изменении серверной логики.

Проверка OpenAPI-контракта имеет отдельное значение. Она подтверждает, что API-маршруты описаны в спецификации и что документация соответствует фактической структуре API. Это важно для сопровождения системы, так как frontend зависит от стабильности серверных маршрутов и формата ответов.

| Подключение к базе данных | Выполнение миграций и сидеров | Таблицы создаются, тестовые данные добавляются |

| Управление растениями | Создание, просмотр, изменение и удаление растения | Данные корректно сохраняются и отображаются |
| Планирование ухода | Создание настроек ухода и отметка задачи | Формируется задача, создается запись в истории |
| Социальные функции | Лайк, совет, подписка, жалоба | Действия выполняются только авторизованным пользователем |
| Администрирование | Обработка жалоб и управление пользователями | Доступ предоставляется только администратору |
| Frontend-сборка | Выполнение сборки проекта | Интерфейс компилируется без ошибок |
| OpenAPI-контракт | Запуск контрактных тестов | API соответствует документации |

Кроме успешных сценариев проверяются ошибочные ситуации. К таким ситуациям относятся ввод неверного пароля, попытка создать растение без обязательного названия, обращение к защищенному API без токена, попытка изменить чужие данные, загрузка некорректного файла и попытка обычного пользователя открыть административную панель. Ожидаемым результатом является отказ в выполнении операции, возврат корректного сообщения об ошибке и сохранение целостности данных.

Проверка негативных сценариев важна, так как она показывает устойчивость системы к неправильным действиям пользователя. Приложение должно не только выполнять корректные операции, но и предсказуемо реагировать на ошибки. В разработанной системе такие ситуации обрабатываются через валидацию, middleware, политики доступа и единый механизм API-ошибок.

## 4.5 Надежность и устойчивость работы

Надежность информационной системы определяется способностью сохранять корректность данных и обеспечивать выполнение основных функций при штатной эксплуатации и при возникновении ошибочных ситуаций. Для рассматриваемого приложения важны надежность хранения пользовательских данных, устойчивость к некорректным запросам, предсказуемая обработка ошибок и возможность восстановления после сбоев.

На уровне базы данных надежность обеспечивается использованием внешних ключей, ограничений уникальности и каскадных правил. Каждое растение связано с владельцем, каждая настройка ухода и запись журнала связана с растением, каждый совет имеет автора и целевой объект. Такие связи позволяют избежать появления изолированных записей, не относящихся к существующим пользователям или растениям. Ограничения уникальности предотвращают повторную регистрацию одного email, дублирование лайков и повторные подписки.

На уровне серверной части надежность повышается за счет валидации входных данных. Перед сохранением информации сервер проверяет типы данных, обязательные поля, допустимые значения и существование связанных записей. Это снижает вероятность попадания в базу данных некорректной информации. Кроме того, часть операций выполняется в транзакциях. Это особенно важно при работе с изображениями, аватарами, жалобами и изменением связанных данных.

При возникновении ошибки API возвращает структурированный ответ. Клиентская часть обрабатывает такой ответ и отображает пользователю сообщение. Благодаря этому ошибка не приводит к неконтролируемому состоянию интерфейса. Например, при отсутствии авторизации пользователь получает требование войти в систему, а при ошибке валидации формы видит причину невозможности сохранения данных.

Устойчивость работы также связана с разделением ответственности между компонентами. Серверная часть отвечает за проверку и сохранение данных, клиентская часть - за отображение и пользовательское взаимодействие. Даже если на клиентской стороне данные формы предварительно проверяются, окончательная проверка выполняется сервером. Это исключает зависимость безопасности и надежности только от браузера пользователя.

Для повышения надежности при эксплуатации рекомендуется предусмотреть регулярное резервное копирование базы данных и каталога загруженных изображений. База данных содержит основную информацию приложения, а файловое хранилище содержит изображения растений и аватары. Потеря любого из этих компонентов приведет к неполному восстановлению системы, поэтому резервное копирование должно охватывать обе части.

## 4.6 Эффективность и производительность

Эффективность приложения определяется скоростью выполнения основных операций, рациональным использованием ресурсов и удобством обработки данных. Для системы учета растений наиболее важны скорость загрузки ленты, списка личных растений, карточки растения, задач ухода и административных списков. Пользователь должен получать результат без заметных задержек, особенно при выполнении типовых действий: просмотр списка, открытие карточки, сохранение формы и отметка задачи выполненной.

На уровне базы данных эффективность обеспечивается индексами. Индексы предусмотрены для часто используемых условий выборки и сортировки: пользователь и дата создания растения, публичность растения и дата создания, комната и название, растение и тип настройки ухода, активность настройки и дата последнего выполнения, растение и дата выполнения ухода, статус жалобы и дата создания. Благодаря этому запросы к ленте, личным растениям, задачам и административным разделам выполняются быстрее.

На уровне backend применяется разделение логики на сервисы. Например, формирование ленты выполняется отдельным сервисом, расчет статистики - отдельным сервисом, работа с изображениями - отдельным сервисом. Такой подход не только упрощает сопровождение, но и позволяет оптимизировать отдельные части приложения без изменения контроллеров и маршрутов.

На уровне frontend эффективность достигается за счет одностраничной архитектуры. После первоначальной загрузки приложения переход между страницами выполняется средствами клиентской маршрутизации, без полной перезагрузки HTML-документа. Данные подгружаются по мере необходимости через API. Это снижает объем повторно передаваемых ресурсов и делает интерфейс более отзывчивым.

Для уменьшения нагрузки на сеть и файловое хранилище изображения растений обрабатываются сервером. При доступности соответствующих средств изображение может быть сжато. Это уменьшает размер файлов, ускоряет загрузку страниц и снижает требования к дисковому пространству.

## 4.7 Понятность и удобство для пользователя

Понятность пользовательского интерфейса является одним из ключевых требований к приложению, поскольку система предназначена для регулярного использования людьми, занимающимися уходом за комнатными растениями. Пользователь должен быстро понимать, где находятся его растения, какие действия необходимо выполнить сегодня, какие растения требуют внимания и как добавить новую запись.

Интерфейс приложения строится на привычных элементах веб-систем: навигации, карточках, формах, кнопках, полях ввода, фильтрах и статусных маркерах. Основная информация о растениях отображается в виде карточек. Такой формат позволяет визуально отделить одну запись от другой и быстро просматривать список. Для задач ухода используются понятные состояния, позволяющие отличать актуальные, просроченные и будущие мероприятия.

Цветовое оформление поддерживает смысловую нагрузку. Зеленые оттенки используются как основные, поскольку они соответствуют предметной области. Красные и желтые элементы применяются для привлечения внимания к проблемным или срочным состояниям. Нейтральные светлые поверхности повышают читаемость текста и не перегружают интерфейс.

## 4.8 Защита информации при эксплуатации

При эксплуатации приложения необходимо обеспечить защиту учетных записей, персональных данных, изображений, истории ухода и административных функций. Основным механизмом защиты является авторизация запросов. Личные данные пользователя доступны только после входа в систему, а административные функции доступны только пользователям с ролью администратора.

Пароли пользователей не должны храниться в открытом виде. В Laravel для этого применяется механизм хеширования. При входе в систему введенный пароль сравнивается с сохраненным хешем, а не с открытым значением. Такой подход снижает риск раскрытия паролей при возможном несанкционированном доступе к базе данных.

Для API применяется токен доступа. После успешной авторизации клиент получает токен и передает его в заголовке каждого защищенного запроса. Сервер проверяет токен и определяет пользователя. Если токен отсутствует или недействителен, запрос не выполняется. Такой механизм особенно удобен для SPA-приложения, где frontend и backend взаимодействуют через REST API.

Разноуровневый доступ реализуется через роли. Обычный пользователь может работать со своими растениями, задачами, профилем и социальной активностью. Администратор имеет дополнительные права, связанные с модерацией жалоб и управлением пользователями. Попытка обычного пользователя открыть административный раздел должна завершаться отказом в доступе.

При эксплуатации также важно ограничивать потенциально опасные операции. Административные маршруты защищены дополнительным ограничением частоты запросов. Действия модератора фиксируются в журнале аудита. Это позволяет отслеживать, какие решения были приняты, каким администратором и в отношении какого объекта.

Для повышения защиты в условиях реальной эксплуатации рекомендуется использовать HTTPS, ограничивать доступ к файлу окружения, отключать режим отладки на production-сервере, регулярно обновлять зависимости и выполнять резервное копирование. ## 4.9 Модифицируемость и сопровождение

Модифицируемость программного продукта определяется тем, насколько удобно вносить изменения, добавлять новые функции и исправлять ошибки без нарушения существующей логики. В разработанном приложении модифицируемость обеспечивается структурным разделением backend и frontend, применением фреймворков, использованием отдельных модулей и наличием тестов.

Серверная часть разделена на контроллеры, модели, запросы валидации, ресурсы, политики и сервисы. Контроллеры принимают запросы и передают данные в соответствующие слои. Модели описывают сущности и связи. Request-классы отвечают за проверку входных данных. Resources формируют единый формат ответа. Policies определяют правила доступа. Services содержат более сложную бизнес-логику, например расчет ленты, статистики, хранение изображений и аудит модерации.

Такое разделение позволяет изменять отдельные части системы с меньшим риском затронуть другие модули. Например, изменение формата отображения растения в API может быть выполнено в ресурсе, изменение правил доступа - в policy, изменение алгоритма рекомендаций - в сервисе ленты. Это повышает профессиональную пригодность проекта к дальнейшему сопровождению.

Клиентская часть также имеет модульную структуру. Страницы отделены от сущностей, виджеты отделены от общих компонентов, а API-логика вынесена в отдельный клиент. Это позволяет повторно использовать элементы интерфейса и состояние. Например, карточка растения может использоваться в ленте и в списке личных растений, а общий API-клиент применяется всеми хранилищами.

Для frontend предусмотрены сборка и статический анализ, что снижает вероятность попадания синтаксических и структурных ошибок в рабочую версию.

## 4.10 Мобильность и масштабируемость

Мобильность приложения выражается в возможности использования системы на разных устройствах без установки специализированного программного обеспечения. Пользователь взаимодействует с приложением через браузер, поэтому доступ возможен с настольного компьютера, ноутбука, планшета или смартфона. Адаптивная структура интерфейса обеспечивает корректное отображение страниц на разных размерах экранов.

Для мобильных устройств предусмотрена отдельная компоновка с нижней навигацией. Такой вариант удобен при использовании смартфона, так как основные разделы находятся в зоне быстрого доступа. На настольных устройствах применяется боковая навигация, позволяющая эффективнее использовать горизонтальное пространство экрана. Наличие двух вариантов оболочки повышает удобство работы и делает приложение более универсальным.

Масштабируемость системы обеспечивается клиент-серверной архитектурой и разделением компонентов. Frontend, backend, база данных и файловое хранилище могут быть размещены и оптимизированы отдельно. При росте числа пользователей возможно увеличение ресурсов сервера, перенос базы данных на отдельный узел, использование кэширования, настройка очередей и отдельное хранение изображений.

В текущей реализации приложение уже содержит предпосылки для масштабирования. API имеет четкую структуру маршрутов, база данных использует индексы, логика сложных выборок вынесена в сервисы, а frontend обращается к серверу через единый клиент. Это позволяет развивать систему постепенно, без полной переработки архитектуры.

## 4.11 Минимизация затрат на сопровождение и поддержку

Минимизация затрат на сопровождение достигается за счет использования распространенных технологий, модульной структуры проекта и автоматизированных средств проверки. Laravel, Vue.js, PostgreSQL и Vite имеют широкую документацию и распространены в практике веб-разработки, что упрощает поиск решений, обновление зависимостей и привлечение специалистов к поддержке проекта.

Приложение не требует установки отдельной клиентской программы на устройства пользователей. Это снижает затраты на распространение и обновление системы. При изменении frontend достаточно обновить сборку на сервере, после чего пользователи получают новую версию через браузер. Серверная часть также может обновляться централизованно.

Использование миграций базы данных упрощает изменение структуры данных. При добавлении новых полей или таблиц изменения фиксируются в виде миграций, которые могут быть применены на разных средах. Это снижает риск ручных ошибок при обновлении базы данных.

Наличие тестов и OpenAPI-спецификации уменьшает затраты на проверку после изменений. Автоматизированные тесты позволяют быстро проверить критически важные сценарии, а спецификация API помогает поддерживать согласованность между frontend и backend. Единый API-клиент на стороне frontend снижает количество мест, где необходимо изменять логику сетевого взаимодействия.
