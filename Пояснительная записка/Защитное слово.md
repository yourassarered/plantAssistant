# Защитное слово по проекту

Уважаемые члены комиссии, вашему вниманию представляется выпускная квалификационная работа на тему разработки веб-приложения клуба любителей растений Plant Assistant.

Актуальность данной темы связана с тем, что уход за комнатными растениями требует регулярности, учета и планирования. На практике информация о растениях часто хранится несистемно: в заметках, мессенджерах, календарях или вовсе не фиксируется. В результате пользователь может забыть о поливе, внесении удобрений, обрезке или других мероприятиях по уходу. Кроме того, при наличии большого количества растений становится сложно отслеживать состояние каждого растения, историю выполненных действий и предстоящие процедуры.

Целью работы являлась разработка веб-приложения, предназначенного для хранения, обработки и визуализации информации о комнатных растениях и мероприятиях по уходу за ними.

Для достижения поставленной цели были решены следующие задачи: был выполнен анализ предметной области, рассмотрены существующие аналоги, определены требования к системе, разработана архитектура приложения, реализованы backend и frontend, предусмотрены механизмы авторизации и разграничения доступа, реализован функционал учета растений и планирования ухода, а также рассмотрены вопросы тестирования, эксплуатации и защиты информации.

Разработанное приложение представляет собой клиент-серверную систему. Пользователь работает с интерфейсом через браузер, а все данные сохраняются на сервере и передаются через REST API. Такой подход позволяет отделить пользовательский интерфейс от серверной логики, упростить сопровождение проекта и обеспечить возможность дальнейшего масштабирования.

Основной функционал приложения включает регистрацию и авторизацию пользователей, управление профилем, добавление и редактирование растений, группировку растений по комнатам, загрузку изображений, настройку регулярного ухода, отображение задач, ведение истории выполненных действий, публичную ленту растений, лайки, советы, подписки и жалобы. Также реализована административная часть, в которой администратор может просматривать пользователей, обрабатывать жалобы, менять роли и контролировать служебные показатели.

Особое внимание в проекте уделено технологической части. Backend реализован на PHP с использованием фреймворка Laravel. Laravel был выбран как надежная основа для построения REST API, поскольку он предоставляет готовые механизмы маршрутизации, контроллеров, моделей, миграций, валидации, авторизации и тестирования. В проекте серверная часть отвечает за обработку запросов, работу с базой данных, проверку прав доступа, формирование JSON-ответов и выполнение бизнес-логики.

Для работы с базой данных используется Eloquent ORM. С его помощью описаны основные сущности предметной области: пользователи, роли, комнаты, растения, изображения, настройки ухода, записи ухода, советы, лайки, подписки, жалобы и журнал действий администратора. Такой подход позволяет работать с данными через модели и связи, а не через разрозненные SQL-запросы. Структура базы данных создается через миграции, что обеспечивает управляемое изменение схемы данных.

В качестве системы управления базами данных используется PostgreSQL. Выбор реляционной СУБД обусловлен тем, что данные проекта имеют четкую структуру и большое количество связей. Растения связаны с пользователями, комнатами, настройками ухода, изображениями и журналом ухода. Пользователи связаны с ролями, подписками, лайками, советами и жалобами. PostgreSQL позволяет обеспечить целостность этих данных с помощью внешних ключей, ограничений уникальности и индексов.

Для защиты API применяется Laravel Sanctum. После входа в систему пользователь получает токен доступа, который передается в последующих запросах. Благодаря этому сервер определяет текущего пользователя и разрешает доступ только к тем данным, с которыми пользователь имеет право работать. Кроме того, в системе предусмотрены роли. Обычный пользователь работает со своими растениями и социальными функциями, а администратор получает доступ к обработке жалоб и управлению пользователями.

Для повышения надежности backend-части используются request-классы валидации, политики доступа и API resources. Request-классы проверяют входные данные до выполнения основной логики. Policies определяют, может ли пользователь выполнить конкретное действие над объектом. API resources формируют единый формат ответов для frontend. Это делает серверную часть более структурированной, безопасной и удобной для сопровождения.

Отдельно следует отметить инфраструктурную часть проекта. Для развертывания может использоваться Docker. Контейнеризация позволяет запускать приложение в одинаковой среде на разных компьютерах и серверах. В контейнеры могут быть вынесены backend, frontend, PostgreSQL и сервер приложений. Это уменьшает зависимость от локальных настроек операционной системы и упрощает запуск проекта при демонстрации или переносе на сервер.

Для повышения производительности backend-части используется подход с RoadRunner. RoadRunner - это высокопроизводительный сервер приложений для PHP, написанный на Go. Его основное отличие от классической схемы запуска PHP заключается в том, что приложение не загружается заново на каждый запрос. Вместо этого RoadRunner запускает пул долгоживущих worker-процессов. Laravel-приложение загружается один раз, остается в памяти и затем обрабатывает множество входящих запросов.

Благодаря RoadRunner ускоряется работа проекта, потому что уменьшаются накладные расходы на повторную инициализацию фреймворка, чтение конфигурации, регистрацию сервисов и подготовку приложения к обработке запроса. Для пользователя это выражается в более быстрой реакции API, особенно на часто используемых страницах: публичной ленте, списке личных растений, задачах ухода и административных списках. Важно подчеркнуть, что ускорение достигается не за счет упрощения бизнес-логики, а за счет более эффективной модели исполнения backend-приложения.

В production-среде RoadRunner требует внимательной настройки. Так как worker-процессы являются долгоживущими, необходимо избегать хранения пользовательских данных в глобальном состоянии, контролировать потребление памяти, настраивать количество worker-процессов и корректно работать с соединениями. При этом такая модель является более производительной для API-приложений, поскольку позволяет обслуживать запросы без постоянного полного перезапуска приложения.

Frontend-часть реализована на Vue.js. Этот фреймворк был выбран, потому что он позволяет создавать интерактивные одностраничные приложения с компонентной структурой. В проекте компоненты используются для карточек растений, форм, кнопок, навигации, задач, календаря, профиля и административной панели. Компонентный подход упрощает развитие интерфейса и позволяет повторно использовать элементы в разных разделах приложения.

Сборка frontend выполняется с помощью Vite. Он обеспечивает быстрый запуск проекта в режиме разработки и формирование оптимизированной production-сборки. Маршрутизация реализована через Vue Router. С его помощью определены страницы ленты, личных растений, карточки растения, формы добавления и редактирования, задач, профиля, публичного профиля пользователя и административной панели. Для защищенных маршрутов настроена проверка авторизации, а для административного раздела дополнительно проверяется роль пользователя.

Для управления состоянием используется Pinia. В проекте состояние разделено по смысловым областям: авторизация, растения, задачи, социальные функции, календарь, статистика и администрирование. Например, хранилище растений отвечает за загрузку ленты, фильтрацию, сортировку, создание и редактирование растений. Хранилище задач формирует мероприятия по уходу на основе настроек растений и отправляет запросы на создание записей в журнале ухода. Хранилище авторизации сохраняет токен и данные текущего пользователя.

Взаимодействие frontend с backend выполняется через единый API-клиент. Он добавляет необходимые заголовки, передает токен авторизации, отправляет JSON-данные или файлы через FormData и обрабатывает ошибки. Благодаря этому сетевое взаимодействие централизовано, а компоненты интерфейса не содержат дублирующей низкоуровневой логики запросов.

Для проверки форм используются VeeValidate и Zod. Эти библиотеки позволяют описывать правила валидации и проверять данные до отправки на сервер. Такой подход улучшает пользовательский опыт, поскольку пользователь быстрее получает сообщение об ошибке. Для графиков и визуализации статистики используется Chart.js совместно с vue-chartjs. Для иконок применяется lucide-vue-next, а для уведомлений - vue-sonner.

Интерфейс приложения адаптирован для разных устройств. На настольных экранах используется боковая навигация, а на мобильных устройствах - нижняя панель. Для работы с адаптивностью применяются возможности Vue и вспомогательные инструменты, включая @vueuse/core. Это позволяет использовать приложение как на компьютере, так и на смартфоне.

Визуально приложение построено в спокойной светлой цветовой схеме с зелеными акцентами, что соответствует тематике ухода за растениями. Карточки растений, статусные маркеры и цветовые обозначения помогают пользователю быстро понять, какие растения требуют внимания и какие задачи необходимо выполнить.

Для контроля качества проекта используются автоматизированные и инструментальные проверки. Backend может проверяться через PHPUnit и OpenAPI-контракт. OpenAPI-спецификация описывает маршруты API и помогает поддерживать согласованность между серверной и клиентской частями. Для PHP-кода используется Laravel Pint. На frontend-стороне применяются ESLint и Prettier, которые обеспечивают проверку и единое форматирование JavaScript и Vue-кода.

В результате работы была создана система, которая объединяет личный учет растений, планирование ухода и социальные функции. Пользователь получает инструмент для хранения информации о растениях, контроля регулярных процедур и взаимодействия с другими участниками. Администратор получает средства модерации и управления пользователями. Технологическая архитектура проекта позволяет развивать приложение дальше: добавлять уведомления, расширенную аналитику, отчеты, экспорт данных, улучшенную систему рекомендаций и production-развертывание с Docker и RoadRunner.

Таким образом, цель работы была достигнута. Разработанное веб-приложение решает поставленные задачи, соответствует требованиям предметной области и демонстрирует применение современных технологий backend- и frontend-разработки. Особое значение в проекте имеют Laravel как основа REST API, Vue.js как основа интерактивного интерфейса, PostgreSQL как надежное хранилище данных, Docker как средство воспроизводимого развертывания и RoadRunner как инструмент повышения производительности серверной части.

## Краткие тезисы для ответа на вопросы

| Вопрос | Краткий ответ |
| --- | --- |
| Зачем используется Laravel? | Для реализации REST API, маршрутизации, бизнес-логики, авторизации, валидации, моделей и миграций |
| Зачем используется Vue.js? | Для построения интерактивного SPA-интерфейса с компонентной структурой |
| Почему PostgreSQL? | Данные имеют реляционную структуру и требуют связей, индексов и ограничений целостности |
| Зачем Laravel Sanctum? | Для токен-based авторизации и защиты API-маршрутов |
| Зачем Docker? | Для воспроизводимого запуска приложения в одинаковой среде на разных машинах |
| Что дает RoadRunner? | Ускоряет backend за счет долгоживущих worker-процессов и отсутствия полной загрузки Laravel на каждый запрос |
| Как обеспечивается безопасность? | Через токены, роли, middleware, policies, валидацию и административный аудит |
| Как обеспечивается качество? | Через PHPUnit, OpenAPI-контракт, Laravel Pint, ESLint и Prettier |
