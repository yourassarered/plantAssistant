# Текст к защитной презентации

## Слайд 1. Главный слайд

Уважаемые члены комиссии, вашему вниманию представляется выпускная квалификационная работа на тему разработки веб-приложения клуба любителей растений Plant Assistant.

## Слайд 2. Актуальность, цель, задачи

Актуальность темы связана с тем, что уход за комнатными растениями требует регулярности, учета и планирования. На практике данные о растениях часто хранятся разрозненно: в заметках, календарях, мессенджерах или вообще не фиксируются. В результате пользователи забывают о поливе, внесении удобрений и других процедурах. Целью работы являлась разработка веб-приложения для хранения, обработки и визуализации информации о растениях и мероприятиях по уходу за ними. Для достижения этой цели были решены задачи анализа предметной области, выбора технологий, проектирования архитектуры, реализации backend и frontend, а также тестирования и оценки работоспособности системы.

## Слайд 3. Основные технологии backend

Основу backend-части составляют PHP 8.3, Laravel 13, PostgreSQL, Eloquent ORM, Laravel Sanctum и PHPUnit. Такой стек был выбран, поскольку он позволяет построить полноценный REST API, обеспечить структурированную работу с данными, безопасную авторизацию и автоматизированную проверку серверной части.

## Слайд 4. PHP и Laravel

Backend реализован на языке PHP с использованием фреймворка Laravel. Laravel предоставляет готовую инфраструктуру для разработки API-приложений: маршрутизацию, контроллеры, модели, middleware, валидацию, политики доступа и тестирование. В рамках проекта именно Laravel обрабатывает входящие запросы, выполняет бизнес-логику и формирует JSON-ответы для frontend.

## Слайд 5. ER-диаграмма, СУБД и ORM

На этом слайде показывается, как организован уровень данных в проекте. В качестве СУБД используется PostgreSQL, поскольку приложение работает со связанной реляционной моделью данных. ER-диаграмма демонстрирует общую структуру этих связей и показывает, что данные о растениях, пользователях, уходе и дополнительных функциях объединены в единую модель. Для работы с базой данных в Laravel используется Eloquent ORM, благодаря которой таблицы и связи представлены в виде моделей. Также важно отметить миграции: они позволяют хранить структуру базы в коде, последовательно изменять схему и воспроизводимо разворачивать проект в другой среде.

## Слайд 6. Laravel Sanctum и авторизация

Для авторизации используется Laravel Sanctum. После входа в систему пользователь получает токен доступа, который передается в запросах как Bearer token. Это позволяет защищать API-маршруты и разграничивать доступ. Также в системе предусмотрены роли обычного пользователя и администратора, а контроль доступа выполняется через middleware и policies. Здесь важно подчеркнуть, что в проекте используется именно Bearer token-авторизация через Sanctum, а не JWT.

## Слайд 7. Docker и RoadRunner

Для развертывания рассматривается использование Docker. Контейнеризация позволяет запускать проект в одинаковой среде на разных машинах и уменьшает зависимость от локальных настроек системы. Для повышения производительности backend применяется RoadRunner. Его особенность состоит в том, что Laravel-приложение загружается один раз и затем обслуживает множество запросов через долгоживущие worker-процессы. За счет этого уменьшаются накладные расходы на повторную инициализацию приложения, и API работает быстрее, особенно на часто вызываемых маршрутах.

## Слайд 8. Vue.js и Vite

Frontend реализован на Vue.js как одностраничное приложение. Vue позволяет строить интерфейс из компонентов и переиспользовать элементы на разных страницах. Для сборки используется Vite, который обеспечивает быстрый запуск проекта в режиме разработки и формирование production-сборки. Маршрутизация организована через Vue Router.

## Слайд 9. Pinia

Для управления состоянием используется Pinia. В проекте созданы отдельные store для авторизации, растений, задач, социальных функций и административной панели. Такой подход позволяет централизовать работу с состоянием, уменьшить дублирование логики и сделать frontend более поддерживаемым.

## Слайд 10. Единый API-клиент

Во frontend используется единый API-клиент. Он отвечает за отправку HTTP-запросов, добавление заголовков, передачу Bearer token, отправку файлов и обработку ошибок. Благодаря этому страницы и компоненты не содержат дублирующей сетевой логики, а взаимодействие с backend остается единообразным.

## Слайд 11. VeeValidate, Zod и другие пакеты frontend

Для проверки форм применяются VeeValidate и Zod. Эти библиотеки используются, например, при работе с формой растения и другими пользовательскими формами. Для графиков и статистики используется Chart.js совместно с vue-chartjs. Для иконок применяется lucide-vue-next, для уведомлений - vue-sonner, а для адаптивной логики и вспомогательных composable используется @vueuse/core. Эти пакеты не являются базовыми для Vue, но делают интерфейс более удобным и функциональным.

## Слайд 12. Тестирование frontend и backend

Для backend применяются PHPUnit и OpenAPI-контракт. Это позволяет проверять API-маршруты, авторизацию, административные сценарии и согласованность документации с фактической реализацией. На frontend используется сборка проекта, статическая проверка через ESLint и контроль форматирования через Prettier. Таким образом обеспечивается как функциональная, так и структурная проверка проекта.

## Слайд 13. Функционал и демонстрация

На завершающем этапе демонстрируется практическая часть проекта. В системе реализованы регистрация и вход, добавление и редактирование растений, настройка ухода, формирование списка задач, публичная лента, социальные функции и административная панель. Демонстрация позволяет показать, что реализованные технологии применены не формально, а обеспечивают работу полноценного пользовательского приложения.

## Завершение выступления

Таким образом, цель работы была достигнута. Разработанное веб-приложение решает задачу систематизированного учета растений и планирования ухода. В проекте применен современный стек технологий: Laravel как основа REST API, PostgreSQL как надежное хранилище данных, Vue.js как основа интерактивного интерфейса, Docker как средство воспроизводимого развертывания и RoadRunner как инструмент повышения производительности backend-части.
