# Анализ проекта Plant Assistant

## Общее описание проекта

Plant Assistant представляет собой полнофункциональное веб-приложение для учета комнатных растений, планирования ухода за ними и организации пользовательского взаимодействия внутри тематического сообщества. Система объединяет инструменты персонального ведения коллекции растений, механизмы контроля регулярных процедур по уходу и социальные возможности для публикации растений, обмена советами и формирования подписок между пользователями.

Проект реализован как современная клиент-серверная система. Серверная часть отвечает за хранение и обработку данных, бизнес-логику, безопасность и предоставление REST API. Клиентская часть реализует интерактивный пользовательский интерфейс, адаптированный под повседневную работу с каталогом растений, задачами ухода и публичной лентой.

## Цель проекта

Целью проекта является разработка удобной информационной системы, которая помогает пользователю систематизировать сведения о комнатных растениях, контролировать мероприятия по уходу, анализировать собственную активность и при необходимости публиковать часть коллекции в общем пространстве для обмена опытом с другими участниками.

## Основные задачи проекта

- обеспечить регистрацию, аутентификацию и безопасную работу пользователей в системе;
- реализовать ведение личного каталога растений с распределением по помещениям;
- предоставить инструменты настройки регулярного ухода и фиксации выполненных действий;
- сформировать ленту публичных растений с возможностью лайков, подписок и советов;
- реализовать механизмы модерации пользовательского контента и административного контроля;
- подготовить проект к сопровождению за счет документированного API, тестирования и понятной архитектурной структуры.

## Текущее состояние проекта

На текущем этапе Plant Assistant представляет собой структурированный full-stack проект с выделенными backend- и frontend-частями.

Серверная часть размещена в корне репозитория и построена на Laravel 13. Она содержит набор API-маршрутов, модели предметной области, слой сервисов, средства валидации, политики доступа, механизмы кэширования и административные инструменты.

Клиентская часть вынесена в отдельный каталог `frontend` и реализована на Vue 3. Она построена как SPA-приложение с маршрутизацией, централизованным управлением состоянием, повторно используемыми UI-компонентами и адаптацией под настольные и мобильные устройства.

Такое разделение позволяет независимо развивать пользовательский интерфейс и серверную логику, сохраняя единый контракт взаимодействия через REST API.

## Архитектурный анализ backend

### Технологический стек backend

В серверной части проекта используются следующие технологии:

- PHP 8.3;
- Laravel 13;
- Laravel Sanctum для токен-ориентированной аутентификации;
- Eloquent ORM для работы с сущностями и связями предметной области;
- PostgreSQL в качестве основной СУБД;
- PHPUnit для функционального тестирования;
- OpenAPI для описания и актуализации API-контракта.

Выбор Laravel обеспечивает проекту четкую организацию слоев, развитую экосистему, встроенные средства безопасности и удобные инструменты для дальнейшего сопровождения.

### Основные backend-модули

Ключевые элементы серверной части представлены следующими модулями:

- `routes/api.php` содержит структуру публичных, авторизованных и административных API-маршрутов;
- `app/Http/Controllers/Api` реализует обработку запросов, связанных с пользователями, растениями, помещениями, расписанием ухода, социальной активностью и модерацией;
- `app/Models` содержит основные сущности системы: пользователей, роли, растения, помещения, настройки ухода, журналы ухода, советы, лайки, подписки, жалобы и медиаданные;
- `app/Services` выделяет прикладную бизнес-логику в отдельный сервисный слой, включая формирование ленты, сводной аналитики, хранение изображений, аудит действий модератора и работу с кэшем;
- `app/Policies` реализует разграничение доступа к объектам системы;
- `app/Http/Requests/Api` и `app/Http/Resources` обеспечивают валидацию входных данных и единообразную структуру API-ответов;
- `database/migrations` и `database/seeders` формируют и заполняют предметную модель базы данных;
- `tests/Feature` содержит набор прикладных тестов для проверки API, контроля доступа, инфраструктуры кэширования, модерации и OpenAPI-контракта.

### Реализованные возможности backend

По текущему состоянию серверная часть поддерживает:

- регистрацию, вход, выход, обновление токена и получение текущего профиля пользователя;
- управление учетной записью, аватаром, ролями и статусом блокировки;
- CRUD-операции для помещений и растений;
- хранение фотографий растений;
- настройку периодичности ухода и ведение журнала выполненных процедур;
- формирование расписания ухода, задач на сегодня, будущих и просроченных мероприятий;
- работу с публичной лентой растений, персональной подборкой, трендовыми и рекомендованными публикациями;
- лайки, подписки и советы между пользователями;
- систему жалоб на растения и советы;
- административную обработку жалоб, блокировку пользователей и просмотр метрик API-трафика.

Отдельного внимания заслуживает наличие сервисов `FeedQueryService` и `DashboardQueryService`, которые централизуют логику выборки и аналитики. Это повышает читаемость кода и создает хороший задел для масштабирования проекта.

## Архитектурный анализ frontend

### Технологический стек frontend

Клиентская часть проекта реализована на основе следующих технологий:

- Vue 3;
- Vue Router;
- Pinia;
- Vite;
- Vee-Validate и Zod для проверки пользовательского ввода;
- Chart.js и `vue-chartjs` для визуализации показателей;
- `@vueuse/core` и `@vueuse/motion` для улучшения интерактивности интерфейса.

Такой стек соответствует современному подходу к разработке SPA-приложений и позволяет строить модульный, расширяемый и отзывчивый интерфейс.

### Структура frontend

Frontend организован по модульному принципу и включает:

- уровень приложения `src/app` с настройкой маршрутизации, оболочек интерфейса и базовых стилей;
- страницы `src/pages`, отражающие основные сценарии работы пользователя;
- доменные сущности `src/entities`, содержащие модели состояния и специализированные визуальные компоненты;
- функциональные модули `src/features`, реализующие отдельные действия пользователя;
- виджеты `src/widgets`, собирающие данные и интерфейс в законченные экранные блоки;
- общий слой `src/shared` с API-клиентом, переиспользуемыми UI-элементами, утилитами и схемами валидации.

### Основные пользовательские сценарии frontend

Интерфейс на текущем этапе поддерживает следующие ключевые страницы и сценарии:

- публичная лента растений;
- просмотр карточки растения;
- просмотр профиля пользователя;
- личный раздел с собственными растениями;
- создание и редактирование растения, параметров ухода и фотографий;
- просмотр и выполнение задач ухода;
- личный профиль пользователя;
- административная страница для контроля пользовательской активности и обращений.

Дополнительно в проекте реализованы адаптивные оболочки `AppShellDesktop` и `AppShellMobile`, что делает интерфейс пригодным как для настольного, так и для мобильного использования. Централизованный API-клиент и набор Pinia-store упрощают поддержку единого состояния приложения и делают взаимодействие с сервером предсказуемым.

## Основные сущности предметной области

Логическая модель проекта опирается на набор взаимосвязанных сущностей:

- `roles` - роли пользователей системы;
- `users` - учетные записи пользователей, включая роль, аватар, ранг и параметры доступа;
- `rooms` - помещения, в рамках которых пользователь группирует растения;
- `plants` - основные объекты учета с характеристиками, владельцем и параметрами публикации;
- `plant_images` - фотографии растений;
- `care_settings` - правила и периодичность выполнения процедур ухода;
- `care_logs` - история фактически выполненных действий;
- `tips` - советы пользователей по уходу;
- `likes` - отметки пользовательского интереса к публичным растениям;
- `follows` - связи подписки между пользователями;
- `reports` - жалобы на пользовательский контент;
- `moderator_audit_logs` - журнал административных и модераторских действий.

Предметная область отражена последовательно и охватывает как личный учет растений, так и социальное взаимодействие внутри системы.

## Входная информация

В качестве входной информации система обрабатывает:

- регистрационные данные пользователя;
- сведения профиля и изображения аватара;
- информацию о помещениях и растениях;
- фотографии растений;
- настройки периодичности ухода;
- записи о выполненных процедурах;
- пользовательские советы, лайки, подписки и жалобы;
- административные действия по модерации и управлению учетными записями.

## Выходная информация

В результате обработки данных система предоставляет пользователю:

- списки и карточки растений;
- персональные и публичные подборки растений;
- расписание ухода, задачи на текущую дату и историю выполненных действий;
- показатели активности, популярности и состояния коллекции;
- профили пользователей и социальные связи;
- результаты модерации и административного контроля;
- структурированные JSON-ответы API для клиентского приложения и тестовых инструментов.

## Сильные стороны текущей реализации

Текущее состояние проекта позволяет выделить несколько сильных сторон:

- четкое разделение серверной и клиентской частей;
- современный и актуальный технологический стек;
- развитая предметная модель, охватывающая как учет растений, так и социальные сценарии;
- наличие административного контура и механизмов модерации;
- использование сервисного слоя, политик доступа и middleware для структурирования бизнес-логики;
- наличие OpenAPI-описания и набора автоматизированных тестов;
- адаптивный интерфейс и модульная организация frontend-части.

В совокупности это позволяет рассматривать Plant Assistant как полноценную информационную систему с хорошим уровнем архитектурной проработки и практической направленностью.

## Материалы проекта, полезные для дальнейшего написания пояснительной записки

Для следующих разделов пояснительной записки особенно полезны:

- `routes/api.php` - для описания функциональной структуры и API;
- `app/Models` и `database/migrations` - для построения логической модели базы данных;
- `app/Services` - для описания прикладной логики и алгоритмов обработки данных;
- `app/Policies` и `app/Http/Middleware` - для раздела по безопасности и разграничению доступа;

- `tests/Feature` и `frontend/scripts/smoke.mjs` - для разделов, связанных с тестированием и проверкой работоспособности.

Он сочетает возможности персонального учета растений, автоматизации регулярного ухода, социальной коммуникации и административного контроля. Архитектура проекта уже демонстрирует системный подход к проектированию, а используемые технологии и структура кода создают надежную основу для дальнейшего развития и оформления последующих разделов пояснительной записки.
