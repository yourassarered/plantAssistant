# Глава 3. Проектирование и программирование

## 3.1 Описание и архитектура разрабатываемого продукта

Разрабатываемое веб-приложение представляет собой информационную систему для учета комнатных растений, планирования мероприятий по уходу и организации взаимодействия пользователей в рамках клуба любителей растений. Система ориентирована на хранение структурированной информации о растениях, ведение истории выполненных действий, формирование задач по уходу, а также на обеспечение социальной активности пользователей через публичную ленту, советы, лайки и подписки.

Проектируемое приложение имеет клиент-серверную архитектуру. Серверная часть отвечает за хранение данных, обработку HTTP-запросов, проверку прав доступа, выполнение бизнес-логики и формирование ответов в формате JSON. Клиентская часть обеспечивает визуальное представление данных, ввод информации пользователем, маршрутизацию между разделами приложения и отправку запросов к серверному API. Такой подход позволяет разделить ответственность между уровнями системы и обеспечить более удобное сопровождение программного продукта.

В качестве серверной технологии применяется фреймворк Laravel. Он используется для построения REST API, работы с базой данных, реализации авторизации, валидации входных данных и разграничения прав доступа. Клиентская часть реализована на Vue.js, что позволяет создавать интерактивный интерфейс одностраничного веб-приложения. Для обмена данными между клиентской и серверной частью применяется архитектурный подход REST, при котором каждая операция выполняется через отдельный HTTP-запрос к определенному маршруту API.

С точки зрения пользователя приложение состоит из набора функциональных разделов. Основными разделами являются публичная лента растений, список личных растений, карточка растения, форма добавления и редактирования растения, задачи по уходу, календарь, профиль пользователя и административная панель. Доступ к отдельным разделам определяется состоянием авторизации пользователя и его ролью. Незарегистрированный пользователь может просматривать публичную ленту и открытые карточки растений, а авторизованный пользователь получает доступ к персональным данным и функциям управления. Администратор дополнительно получает доступ к разделу модерации и управлению пользователями.

Основным объектом предметной области является растение. Для растения сохраняются название, дата посадки, высота, признак публичности, принадлежность пользователю и связь с комнатой. Дополнительно растение может иметь изображения, настройки ухода, историю выполненных действий, советы и лайки. Комната используется для группировки растений внутри учетной записи пользователя. Настройки ухода определяют периодичность выполнения процедур, например полива, внесения удобрений, обрезки или поворота растения. История ухода фиксирует факт выполнения конкретной процедуры в определенный момент времени.

Раздел авторизации предназначен для регистрации пользователей, входа в систему, выхода из учетной записи и получения данных текущего пользователя. Раздел управления растениями обеспечивает создание, просмотр, редактирование и удаление записей о растениях. Раздел планирования ухода формирует задачи на основе заданных интервалов и позволяет фиксировать выполненные действия. Социальный раздел предоставляет возможность просматривать публичные растения, ставить лайки, оставлять советы и подписываться на других пользователей. Административный раздел используется для обработки жалоб, изменения ролей пользователей и просмотра служебных метрик.

При проектировании внешнего вида приложения учитывалась необходимость создать спокойный и понятный интерфейс, соответствующий тематике ухода за растениями. Визуальная система основана на светлом фоне, белых поверхностях и зеленых акцентах. Зеленый цвет используется как основной тематический цвет, так как он ассоциируется с растениями, ростом и уходом. Дополнительные цвета применяются для обозначения состояний: желтый используется для предупреждающих элементов, красный - для просроченных или критических состояний, синий - для нейтральных информационных элементов. Такой подход позволяет пользователю быстрее воспринимать состояние растений и задач.

В качестве основного шрифта используется семейство Trebuchet MS с резервным вариантом Segoe UI. Выбор шрифта обусловлен необходимостью обеспечить читаемость интерфейса на разных устройствах и сохранить визуальную легкость приложения. В интерфейсе применяются карточки растений, панели фильтрации, кнопки действий, поля ввода, бейджи статусов, нижняя мобильная навигация и боковая навигация для настольной версии. Интерфейс адаптирован под мобильные и настольные экраны, что повышает доступность системы для пользователей с разными устройствами.

Страница личных растений строится вокруг карточек растений. Каждая карточка содержит краткую информацию о растении, изображение, комнату, показатели ухода и визуальные маркеры состояния. Форма добавления и редактирования растения является одной из ключевых форм приложения. Она содержит поля для ввода названия, комнаты, даты посадки, высоты, настроек ухода и изображения. На основе введенных данных формируется запрос к серверу. При создании растения дополнительно создаются настройки ухода, а при загрузке изображения файл отправляется в отдельный API-маршрут. Форма является примером CRUD-функциональности, так как она используется для создания и изменения основной сущности системы.

Входной информацией для системы являются регистрационные данные, данные профиля, сведения о растениях, параметры комнат, настройки ухода, записи о выполненных действиях, изображения, советы, лайки, подписки и жалобы. Выходной информацией являются списки растений, карточки растений, календарь ухода, задачи на сегодня, история выполненных процедур, публичная лента, профили пользователей, административные списки жалоб и пользователей, а также сообщения о результатах операций.

Логическая модель базы данных строится вокруг пользователя и связанных с ним объектов. Пользователь имеет роль, может создавать комнаты и растения. Растения связаны с настройками ухода, журналом ухода, изображениями, лайками и советами. Пользователи могут подписываться друг на друга, а жалобы могут относиться к разным типам объектов. Такая модель обеспечивает как персональный учет, так и социальные функции приложения.

Таблица `roles` содержит идентификатор роли и ее уникальное наименование. В системе используются роли `user` и `admin`. Роль `user` предназначена для обычного пользователя, а роль `admin` предоставляет доступ к административным функциям. Наличие отдельной таблицы ролей позволяет расширять модель доступа без изменения структуры таблицы пользователей.

Таблица `users` содержит данные учетной записи. К основным полям относятся имя, адрес электронной почты, пароль, идентификатор роли, ранг пользователя и путь к аватару. Поле электронной почты имеет ограничение уникальности, что исключает регистрацию нескольких учетных записей с одним адресом. Пароль хранится не в открытом виде, а в виде хеша, что повышает уровень защиты пользовательских данных.

Таблица `rooms` используется для группировки растений по помещениям. Каждая комната принадлежит конкретному пользователю. При удалении пользователя связанные с ним комнаты удаляются каскадно. Такая связь обеспечивает целостность данных и предотвращает появление комнат, не принадлежащих ни одному пользователю.

Таблица `plants` является центральной таблицей предметной области. Она содержит название растения, дату посадки, высоту, признак публичности, идентификатор владельца и идентификатор комнаты. Связь с пользователем является обязательной, так как каждое растение должно иметь владельца. Связь с комнатой является необязательной, поскольку пользователь может создать растение без указания помещения. При удалении комнаты значение ссылки на комнату становится пустым, что позволяет сохранить растение.

Таблица `care_settings` определяет правила регулярного ухода. Для каждой настройки указывается растение, тип процедуры, интервал в днях, признак активности и дата последнего выполнения. Тип процедуры ограничен фиксированным набором значений: полив, удобрение, обрезка и поворот. Наличие интервала позволяет системе рассчитывать дату следующего действия и формировать задачи для пользователя.

Таблица `care_logs` хранит историю фактически выполненных действий. В ней фиксируются растение, тип процедуры, дата выполнения и комментарий. Эта таблица используется для отображения истории ухода и обновления состояния растения после выполнения задачи. Связь с растением обеспечивает возможность быстро получить журнал ухода для конкретной карточки.

Таблица `plant_images` содержит путь к файлу изображения, исходное имя файла и размер. Изображение связано с растением. Физическое хранение файла осуществляется в файловом хранилище приложения, а в базе данных сохраняется только служебная информация, необходимая для доступа и управления файлом.

Таблица `tips` предназначена для хранения советов. Совет связан с растением и автором. Для совета хранится текст и статус обработки: ожидает рассмотрения, принят или отклонен. Такой механизм позволяет владельцу растения управлять поступающими рекомендациями и учитывать полезные советы.

Таблица `likes` хранит лайки растений. Для пары пользователь-растение установлено ограничение уникальности, поэтому один пользователь не может поставить несколько лайков одному и тому же растению. Таблица `follows` работает по аналогичному принципу и хранит подписки между пользователями. Уникальность пары подписчик-пользователь исключает дублирование подписки.

Таблица `reports` используется для жалоб. Жалоба содержит автора, тип целевого объекта, идентификатор целевого объекта, причину, подробности, статус обработки, комментарий администратора и данные о проверяющем. Использование полей `target_type` и `target_id` позволяет применять одну таблицу для жалоб на разные сущности, например на растение или совет. Таблица `moderator_audit_logs` фиксирует административные действия и используется для контроля действий модератора.

Для ускорения работы системы в базе данных используются индексы. Индексы добавлены для выборок растений пользователя, публичных растений, настроек ухода, истории ухода, советов, лайков, подписок, жалоб и изображений. Это имеет значение для страниц ленты, личных растений, административной панели и календаря ухода, так как данные разделы выполняют выборки с сортировкой и фильтрацией.

## 3.2 Разработка внутренней структуры и программирование

Физическая структура проекта отражает разделение приложения на серверную и клиентскую части. Корневая часть проекта содержит Laravel-приложение, конфигурационные файлы, маршруты, миграции, тесты и публичные ресурсы. Клиентская часть размещена в отдельной папке `frontend`, что позволяет независимо развивать интерфейс и серверное API.

Структура проекта представлена на рисунке 3.7.

[Рисунок 3.7 - Физическая структура проекта: корень Laravel-приложения, папка app, routes, database, tests, public и отдельная папка frontend]

Папка `app` содержит основной серверный код. Внутри нее размещены модели, контроллеры, запросы валидации, ресурсы API, политики доступа, middleware и сервисные классы. Папка `routes` содержит описание маршрутов. Наиболее значимым файлом является `routes/api.php`, где определены публичные, защищенные и административные маршруты. Папка `database` содержит миграции и сидеры, используемые для создания и первичного наполнения базы данных. Папка `tests` содержит автоматизированные тесты, проверяющие корректность работы API и соответствие OpenAPI-контракту.

Клиентская часть в папке `frontend` построена как SPA-приложение. В папке `src/app` размещены глобальные настройки приложения, маршрутизация, layout и стили. Папка `src/pages` содержит страницы, соответствующие основным маршрутам. Папка `src/entities` содержит хранилища состояния и компоненты предметных сущностей. Папка `src/features` используется для отдельных пользовательских действий, а папка `src/shared` содержит общие компоненты, API-клиент, мапперы и вспомогательные функции.

Для хранения состояния на клиентской стороне используется Pinia. Состояние авторизации хранит токен, данные текущего пользователя и признак инициализации. Состояние растений содержит список загруженных растений, параметры фильтрации, сортировки и выбранный режим ленты. Состояние задач формируется на основе настроек ухода и используется для отображения текущих, просроченных и будущих действий. Отдельные состояния применяются для социальных функций, административных данных, календаря и статистики.

Общий механизм обращения к API реализован в файле `frontend/src/shared/api/client.js`. Данный модуль формирует HTTP-запросы, добавляет необходимые заголовки и обрабатывает ошибки. Если пользователь авторизован, в запрос автоматически добавляется заголовок авторизации с Bearer-токеном. Благодаря этому компоненты интерфейса не работают напрямую с низкоуровневой логикой `fetch`, а используют единый клиент.

Фрагмент реализации общего API-клиента приведен ниже.

```javascript
async request(path, options = {}) {
    const isFormData = options.body instanceof FormData;
    const response = await fetch(`${API_BASE_URL}${path}`, {
        ...options,
        headers: {
            Accept: "application/json",
            ...(options.body && !isFormData ? { "Content-Type": "application/json" } : {}),
            ...(this.token ? { Authorization: `Bearer ${this.token}` } : {}),
            ...options.headers,
        },
        body: isFormData ? options.body : options.body ? JSON.stringify(options.body) : undefined,
    });

    const payload = response.status === 204 ? null : await response.json().catch(() => null);

    if (!response.ok) {
        const validationMessage = payload?.errors
            ? Object.values(payload.errors).flat().join(" ")
            : "";
        const error = new Error(
            validationMessage || payload?.message || `API request failed: ${response.status}`,
        );
        error.status = response.status;
        error.payload = payload;
        throw error;
    }

    return payload;
}
```

Представленный фрагмент показывает, что обработка запросов выполняется централизованно. При отправке обычных JSON-данных тело запроса преобразуется в строку, а при отправке изображения используется объект `FormData`. В случае ошибки создается объект исключения с текстом ошибки и статусом ответа. Такой механизм упрощает обработку ошибок в интерфейсе и обеспечивает единообразное поведение всех экранов.

Если пользователь не имеет роли администратора, доступ к административной странице запрещается и выполняется переход в ленту.

```javascript
async addPlant(values, imageFile = null) {
    if (!apiClient.token) {
        throw new Error("Нужно войти, чтобы добавлять растения.");
    }

    let roomId = null;
    if (values.room) {
        const roomsPayload = await apiClient.get("/rooms?per_page=100");
        const existingRoom = unwrapApiCollection(roomsPayload).find(
            (room) => room.name.toLowerCase() === values.room.toLowerCase(),
        );

        if (existingRoom) {
            roomId = existingRoom.id;
        } else {
            const roomPayload = await apiClient.post("/rooms", { name: values.room });
            roomId = roomPayload.data?.id || roomPayload.id;
        }
    }

    const plantPayload = await apiClient.post("/plants", mapPlantFormToApi(values, roomId));
    const plant = plantPayload.data || plantPayload;
    const careSettings = mapCareSettingsFromForm(values);

    await Promise.all(
        careSettings.map((setting) => apiClient.post(`/plants/${plant.id}/care-settings`, setting)),
    );

    if (imageFile) {
        const formData = new FormData();
        formData.append("image", imageFile);
        await apiClient.postForm(`/plants/${plant.id}/images`, formData);
    }

    await this.loadPlants();
    return plant;
}
```

В приведенном фрагменте реализована последовательная обработка данных формы. Код демонстрирует, что создание растения не ограничивается одной записью в таблице `plants`. В процессе сохранения участвуют связанные сущности: комната, настройки ухода и изображение. ## 3.3 Обеспечение безопасности информации

Безопасность информации является важной частью проектирования приложения, поскольку система хранит персональные данные пользователей, сведения о растениях, изображения, историю действий и социальные взаимодействия. Основными угрозами для такой системы являются несанкционированный доступ к чужим данным, подмена идентификаторов объектов в запросах, отправка некорректных данных, удаление чужих записей, загрузка нежелательных файлов, злоупотребление административными функциями и потеря целостности данных.

Для защиты от несанкционированного доступа в приложении используется механизм аутентификации на основе Laravel Sanctum. При успешном входе пользователь получает токен доступа. Данный токен сохраняется на клиентской стороне и передается в последующих запросах в заголовке `Authorization`. Сервер проверяет токен и определяет текущего пользователя. Запросы к персональным данным выполняются только для авторизованных пользователей.

Общая схема авторизации представлена на рисунке 3.12.

[Рисунок 3.12 - Схема авторизации: ввод email и пароля, проверка учетных данных на сервере, выдача токена, отправка токена в защищенных запросах]

Разграничение доступа реализуется на нескольких уровнях. Первый уровень связан с маршрутизацией: публичные маршруты доступны всем пользователям, защищенные маршруты требуют авторизации, а административные маршруты требуют роль `admin`. Второй уровень связан с политиками доступа, которые проверяют право пользователя выполнять действие над конкретным объектом. Третий уровень обеспечивается валидацией входных данных, исключающей сохранение некорректной или потенциально опасной информации.

В системе предусмотрены две основные роли: обычный пользователь и администратор. Обычный пользователь может управлять собственными растениями, комнатами, настройками ухода, профилем и социальной активностью. Администратор получает дополнительные возможности: просмотр жалоб, обработка жалоб, изменение ролей пользователей, удаление учетных записей и просмотр метрик трафика. Такое разделение соответствует требованию разноуровневого доступа к информации.

Административные маршруты дополнительно ограничены middleware `admin` и механизмом rate limiting. Это снижает риск злоупотребления административными операциями и ограничивает частоту выполнения чувствительных действий. Решения администратора по жалобам фиксируются в журнале аудита. В журнал записываются действие, целевой объект, дополнительные данные, IP-адрес и пользовательский агент. Наличие аудита позволяет анализировать действия модератора и повышает контролируемость системы.

Для защиты данных от некорректного ввода используются request-классы Laravel и встроенная валидация. Например, при создании растения проверяется обязательность названия, корректность даты, числовой формат высоты и существование комнаты. При отправке жалобы проверяется причина и дополнительные сведения. При загрузке изображений проверяются параметры файла. Такой подход уменьшает вероятность сохранения неполных или ошибочных данных.

Целостность данных обеспечивается внешними ключами и каскадными правилами. При удалении пользователя удаляются связанные комнаты, растения и другие зависимые записи, если это предусмотрено логикой таблиц. При удалении комнаты растение не удаляется, а ссылка на комнату становится пустой. Это соответствует предметной логике: растение может существовать без привязки к комнате. Ограничения уникальности применяются для лайков и подписок, что предотвращает дублирование записей.

Для защиты изображений используется выделенный сервис хранения файлов. Сервер сохраняет файл в определенном каталоге, при возможности выполняет сжатие изображения, а в базе данных хранит путь и служебные параметры. Удаление изображения выполняется через сервис, что позволяет синхронизировать состояние базы данных и файлового хранилища.

Дополнительной мерой безопасности является принудительное формирование JSON-ответов для API. Это обеспечивает единообразную обработку ошибок на клиентской стороне. При возникновении ошибки авторизации или валидации frontend получает структурированный ответ и может корректно отобразить сообщение пользователю. Такой подход повышает предсказуемость поведения системы.

Таким образом, безопасность приложения обеспечивается сочетанием аутентификации, авторизации, валидации, разграничения ролей, контроля административных действий и ограничений на уровне базы данных. Применение нескольких взаимодополняющих механизмов позволяет снизить риск несанкционированного доступа и повысить надежность хранения информации.
