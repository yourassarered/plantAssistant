# Анализ проекта Plant Assistant

## Общее описание проекта

Проект представляет собой веб-приложение для учета комнатных растений и организации ухода за ними. Система позволяет пользователю хранить информацию о своих растениях, группировать их по комнатам, задавать регулярные мероприятия по уходу, отмечать выполненные действия и просматривать задачи на ближайшие даты.

Помимо личного учета, в проекте реализована социальная часть: публичная лента растений, лайки, советы другим пользователям, подписки на профили и жалобы на некорректный контент. Для администраторов предусмотрены функции модерации жалоб, управления пользователями и просмотра метрик API-трафика.

Приложение разделено на две основные части:

- backend на Laravel, отвечающий за API, бизнес-логику, хранение данных, авторизацию и модерацию;
- frontend на Vue 3, отвечающий за пользовательский интерфейс, маршрутизацию, состояние экранов и взаимодействие с API.

## Цель проекта

Цель проекта - разработать веб-приложение клуба любителей растений, которое обеспечивает удобное хранение, обработку и визуализацию информации о комнатных растениях, мероприятиях по уходу и социальной активности пользователей.

## Основные задачи проекта

- реализовать регистрацию, авторизацию и разграничение прав пользователей;
- обеспечить хранение данных о пользователях, комнатах, растениях, изображениях, настройках ухода и истории выполненных действий;
- реализовать CRUD-операции для растений, комнат, настроек ухода и пользовательского профиля;
- сформировать механизм планирования ухода на основе периодичности процедур;
- предоставить пользователю список задач на сегодня, просроченные задачи и календарное представление ухода;
- реализовать публичную ленту растений, лайки, подписки и советы;
- добавить систему жалоб и административную модерацию;
- обеспечить защиту API с помощью токенов, ролей, policy-авторизации и административных ограничений;
- подготовить проект к тестированию через API-тесты, OpenAPI-контракт и smoke-проверки frontend.

## Анализ backend

Backend расположен в корневой части проекта и построен на Laravel 13. Основная задача серверной части - предоставить REST API для frontend-приложения и обеспечить надежную работу с данными.

### Технологии backend

- PHP 8.3+;
- Laravel 13;
- Laravel Sanctum для token-based аутентификации;
- Eloquent ORM для работы с моделями и связями;
- PostgreSQL согласно README и настройкам проекта;
- PHPUnit для feature- и contract-тестов;
- OpenAPI/Swagger для описания API;
- Laravel policies, middleware и form requests для авторизации, фильтрации и валидации.

В существующем файле `Введение.md` указана MySQL, но README проекта и `.env.example` ориентируют систему на PostgreSQL. При дальнейшем написании пояснительной записки это нужно привести к одному варианту, вероятнее всего к PostgreSQL.

### Основные backend-модули

- `routes/api.php` - описание публичных, защищенных и административных API-маршрутов.
- `app/Http/Controllers/Api` - контроллеры обработки запросов: авторизация, пользователи, растения, комнаты, уход, лента, советы, лайки, подписки, жалобы, администрирование.
- `app/Models` - предметные модели: `User`, `Role`, `Plant`, `Room`, `CareSetting`, `CareLog`, `Tip`, `Like`, `Follow`, `Report`, `PlantImage`, `ModeratorAuditLog`.
- `app/Http/Requests/Api` - классы валидации входных данных.
- `app/Http/Resources` - форматирование API-ответов.
- `app/Policies` - правила доступа к сущностям.
- `app/Services` - сервисы для сложной логики: лента, дашборд, кэширование, хранение изображений, аудит модерации.
- `database/migrations` - структура базы данных.
- `database/seeders` - начальное заполнение справочников и тестовых данных.
- `tests/Feature` - функциональные тесты API, модерации, кэширования, публичного чтения и OpenAPI-контракта.

### Основные сущности backend

- `roles` - роли пользователей (`user`, `admin`);
- `users` - учетные записи пользователей, ранг, аватар, привязка к роли;
- `rooms` - комнаты пользователя для группировки растений;
- `plants` - растения пользователя: название, дата посадки, высота, публичность, владелец, комната;
- `plant_images` - изображения растений;
- `care_settings` - настройки регулярного ухода: тип процедуры, интервал, активность, дата последнего выполнения;
- `care_logs` - история фактически выполненных процедур;
- `tips` - советы пользователей к публичным растениям;
- `likes` - отметки понравившихся растений;
- `follows` - подписки между пользователями;
- `reports` - жалобы на растения или советы;
- `moderator_audit_logs` - журнал действий модератора;
- `personal_access_tokens`, `sessions`, `cache`, `jobs` - инфраструктурные таблицы Laravel.

### Функциональность backend

Публичные маршруты позволяют читать ленту, публичные растения, советы и количество лайков. Защищенные маршруты требуют токен Sanctum и позволяют пользователю управлять личными растениями, комнатами, настройками ухода, историей ухода, лайками, советами, подписками и профилем. Административные маршруты дополнительно защищены middleware `admin` и rate limit `admin-actions`.

Серверная часть содержит несколько важных механизмов:

- проверка прав доступа через policies;
- валидация данных через request-классы;
- автоматическое форматирование ответов через resources;
- загрузка и сжатие изображений через `ImageStorageService`;
- расчет ленты и рекомендаций через `FeedQueryService`;
- расчет статистики пользователя через `DashboardQueryService`;
- аудит модераторских действий через `ModeratorAuditLogger`;
- индексы в БД для ускорения выборок по растениям, лайкам, жалобам, советам и подпискам.

## Анализ frontend

Frontend расположен в папке `frontend` и реализован как отдельное Vue 3 SPA-приложение. Он обращается к Laravel API через общий HTTP-клиент и отображает данные в виде адаптивного интерфейса для desktop и mobile.

### Технологии frontend

- Vue 3;
- Vite;
- Vue Router;
- Pinia и `pinia-plugin-persistedstate`;
- VeeValidate и Zod для валидации форм;
- Chart.js и `vue-chartjs` для графиков;
- `lucide-vue-next` для иконок;
- `vue-sonner` для уведомлений;
- ESLint и Prettier для контроля качества кода.

### Архитектура frontend

Структура frontend близка к feature/entity-подходу:

- `src/app` - точка сборки приложения, layout, router и глобальные стили;
- `src/pages` - страницы приложения: лента, мои растения, задачи, календарь, профиль, страница растения, редактирование растения, профиль пользователя, админка;
- `src/entities` - предметные состояния и UI-компоненты: растения, задачи, календарь, авторизация, социальные функции, администрирование, дашборд;
- `src/features` - отдельные пользовательские действия: фильтрация растений, отметка задачи выполненной;
- `src/widgets` - крупные виджеты: список растений, задачи на сегодня, календарь, навигация;
- `src/shared` - общий API-клиент, мапперы, UI-компоненты, валидация, утилиты дат и типов ухода.

### Основные экраны frontend

- `/feed` - публичная и персональная лента растений;
- `/my-plants` - список личных растений пользователя;
- `/plants/:id` - подробная карточка растения;
- `/add-plant` и `/plants/:id/edit` - форма создания и редактирования растения;
- `/tasks` - задачи по уходу;
- `/users/:id` - публичный профиль пользователя;
- `/profile` - авторизация, регистрация и управление профилем;
- `/admin` - административная панель;
- `/:pathMatch(.*)*` - страница 404.

В router есть защита маршрутов: страницы личных растений, задач, добавления/редактирования и админки требуют авторизации, а `/admin` дополнительно требует роль администратора.

### Состояние и API frontend

Взаимодействие с API централизовано в `src/shared/api/client.js`. Клиент добавляет `Accept: application/json`, `Content-Type` для JSON-запросов и `Authorization: Bearer <token>`, если пользователь авторизован. Токен хранится в `localStorage`.

Основные Pinia-store:

- `auth.store.js` - вход, регистрация, загрузка текущего пользователя, выход, профиль и аватар;
- `plant.store.js` - загрузка ленты и личных растений, фильтры, сортировка, создание, обновление и удаление растений;
- `task.store.js` - формирование задач из настроек ухода, отметка выполнения через создание `care_logs`;
- `social.store.js` - лайки, советы, жалобы и социальные данные растения;
- `admin.store.js` - пользователи, жалобы, трафик, смена ролей и модерация;
- `dashboard.store.js` - обзор, активность и статистика здоровья растений;
- `calendar.store.js` - состояние календарного представления.

### Интерфейс и визуальный стиль

Интерфейс ориентирован на спокойную рабочую веб-систему для учета растений. В CSS-токенах используются светлый фон, белые поверхности, зеленые акценты, желтый/красный/синий для состояний и событий. Основной шрифт - `Trebuchet MS` с fallback на `Segoe UI`. Приложение имеет разные оболочки для desktop и mobile, нижнюю мобильную навигацию и боковую desktop-навигацию.

Для пояснительной записки это можно использовать в разделе 3.1 при обосновании выбора цветов, шрифтов и основных элементов интерфейса.

## Входная и выходная информация

### Входная информация

- регистрационные данные пользователя: имя, email, пароль;
- данные профиля и аватар;
- данные растения: название, дата посадки, высота, комната, публичность, изображение;
- настройки ухода: тип процедуры, интервал выполнения, активность;
- записи ухода: тип процедуры, дата выполнения, комментарий;
- социальные действия: лайк, подписка, совет, жалоба;
- административные действия: смена роли, решение по жалобе, комментарий модератора.

### Выходная информация

- списки растений и карточки растений;
- публичная лента и персональные подборки;
- расписание ухода, задачи на сегодня, просроченные и будущие задачи;
- история ухода по растениям;
- счетчики лайков, советы и статусы советов;
- профиль пользователя, подписчики и подписки;
- административные списки пользователей, жалоб и метрики трафика;
- уведомления об ошибках, успешных действиях и результатах валидации.
