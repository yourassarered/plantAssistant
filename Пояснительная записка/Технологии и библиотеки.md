## Текст для защиты: backend

Серверная часть проекта реализована на языке PHP с использованием фреймворка Laravel. Данный фреймворк был выбран, поскольку он предоставляет готовую инфраструктуру для разработки веб-приложений и REST API. В рамках проекта Laravel используется как основа backend-части.

Для работы с данными применяется Eloquent ORM. С его помощью описаны основные сущности системы: пользователи, роли, комнаты, растения, изображения растений, настройки ухода, история ухода, советы, лайки, подписки, жалобы и журнал действий администратора. Связи между сущностями реализованы через модели Laravel, что позволяет работать с данными в объектно-ориентированном виде и поддерживать логическую структуру приложения. Структура базы данных создается через миграции, а начальные данные формируются с помощью сидеров. Такой подход упрощает развертывание проекта на новом рабочем месте и обеспечивает контролируемое изменение структуры базы данных.

В качестве системы управления базами данных используется PostgreSQL.

Для защиты API используется Laravel Sanctum. (покорочё ненадо пояснять про jwt одним предложением)

Для развертывания backend-части также используется контейнеризация с использованием Docker. Docker позволяет запускать приложение в одинаковой среде независимо от операционной системы разработчика или сервера. Это упрощает установку проекта, снижает количество ошибок, связанных с различиями окружения, и делает процесс запуска более воспроизводимым.

Для повышения производительности серверной части может использоваться RoadRunner в связке с Laravel Octane. RoadRunner является высокопроизводительным сервером приложений для PHP, написанным на Go. В отличие от классической модели, при которой PHP-приложение фактически инициализируется заново для каждого запроса, RoadRunner запускает пул долгоживущих worker-процессов. Laravel-приложение загружается один раз, остается в памяти и далее обслуживает множество запросов. Благодаря этому уменьшаются накладные расходы на повторную загрузку фреймворка, регистрацию сервисов, чтение конфигурации и подготовку окружения.

Применение RoadRunner ускоряет backend не за счет изменения бизнес-логики, а за счет изменения модели исполнения PHP-приложения. Запросы к API обрабатываются быстрее, поскольку готовые worker-процессы уже содержат загруженное приложение. Особенно заметным это становится для часто вызываемых маршрутов: публичной ленты, списка растений, задач ухода и административных списков. При этом для стабильной работы такого режима необходимо учитывать особенности долгоживущих процессов: не хранить пользовательские данные в глобальном состоянии, следить за памятью, корректно закрывать соединения и настраивать количество worker-процессов в соответствии с ресурсами сервера.

Качество backend-части контролируется с помощью PHPUnit, OpenAPI-контракта и Laravel Pint. PHPUnit используется для автоматизированной проверки API-сценариев. OpenAPI-спецификация описывает доступные маршруты и формат данных, что важно для согласованной работы frontend и backend. Laravel Pint применяется для поддержания единого стиля PHP-кода. В результате серверная часть проекта построена как полноценное REST API, обеспечивающее хранение данных, безопасность, бизнес-логику, производительный запуск через RoadRunner и интеграцию с клиентским приложением.

## Текст для защиты: frontend

Клиентская часть проекта реализована на Vue.js. Данный фреймворк был выбран, поскольку он позволяет создавать интерактивные одностраничные приложения с компонентной структурой.

Для сборки и запуска frontend-части используется Vite. Он обеспечивает быстрый запуск dev-сервера, удобную разработку Vue-компонентов и формирование production-сборки. Маршрутизация реализована через Vue Router. С его помощью определены основные страницы приложения и настроена проверка доступа к защищенным маршрутам. Если пользователь не авторизован, он перенаправляется на страницу профиля, а административный раздел доступен только пользователю с ролью администратора.

Состояние приложения хранится в Pinia. В проекте созданы отдельные хранилища для авторизации, растений, задач, социальной активности, календаря, статистики и административных данных. Такой подход позволяет отделить бизнес-состояние интерфейса от отдельных компонентов.

Для взаимодействия с backend используется единый API-клиент. Он формирует HTTP-запросы, добавляет заголовки, передает Bearer-токен авторизованного пользователя, обрабатывает JSON-ответы и ошибки. Благодаря этому компоненты frontend не работают напрямую с низкоуровневой логикой запросов, а используют единый механизм обращения к API. Это упрощает сопровождение и делает поведение приложения более единообразным.

Для работы с формами применяются VeeValidate и Zod. Они используются для проверки вводимых данных на клиентской стороне, что помогает пользователю быстрее исправлять ошибки до отправки запроса на сервер. Для визуализации статистики используется Chart.js вместе с vue-chartjs. Иконки интерфейса реализованы через lucide-vue-next, а уведомления - через vue-sonner. Дополнительные библиотеки, такие как @vueuse/core, floating-vue, embla-carousel-vue и clsx, используются для адаптивности, всплывающих элементов, карусельных интерфейсов и удобной работы с CSS-классами.

Качество frontend-кода поддерживается с помощью ESLint и Prettier. ESLint проверяет JavaScript и Vue-компоненты на ошибки и нарушения правил, а Prettier обеспечивает единое форматирование. В результате клиентская часть проекта представляет собой современное SPA-приложение, которое обеспечивает удобный интерфейс для работы с растениями, задачами ухода, социальной лентой и административными функциями, а также взаимодействует с backend через REST API.
