## Текст для защиты: backend

Серверная часть проекта реализована на языке PHP с использованием фреймворка Laravel. Данный фреймворк был выбран, поскольку он предоставляет готовую инфраструктуру для разработки веб-приложений и REST API: маршрутизацию, контроллеры, модели, миграции, валидацию запросов, обработку ошибок, авторизацию и тестирование. В рамках проекта Laravel используется как основа backend-части, которая принимает запросы от клиентского приложения, выполняет бизнес-логику, обращается к базе данных и возвращает ответы в формате JSON.

Для работы с данными применяется Eloquent ORM. С его помощью описаны основные сущности системы: пользователи, роли, комнаты, растения, изображения растений, настройки ухода, история ухода, советы, лайки, подписки, жалобы и журнал действий администратора. Связи между сущностями реализованы через модели Laravel, что позволяет работать с данными в объектно-ориентированном виде и поддерживать логическую структуру приложения. Структура базы данных создается через миграции, а начальные данные формируются с помощью сидеров. Такой подход упрощает развертывание проекта на новом рабочем месте и обеспечивает контролируемое изменение структуры базы данных.

В качестве системы управления базами данных используется PostgreSQL. Данная СУБД подходит для проекта, поскольку данные приложения имеют четкую реляционную структуру и требуют поддержки внешних ключей, индексов и ограничений уникальности. В базе данных хранятся пользовательские аккаунты, растения, настройки ухода, журнал выполненных действий и социальные связи. Индексы применяются для ускорения выборок по публичной ленте, личным растениям, жалобам, лайкам и подпискам.

Для защиты API используется Laravel Sanctum. После успешной авторизации пользователь получает токен доступа, который передается в последующих запросах. Сервер проверяет токен и определяет текущего пользователя. Это позволяет закрыть персональные маршруты и предоставить доступ только авторизованным пользователям. Дополнительно в проекте используется разграничение доступа по ролям: обычный пользователь работает со своими растениями и социальными функциями, а администратор получает доступ к обработке жалоб, управлению пользователями и просмотру метрик.

Для развертывания backend-части также рассматривается контейнеризация с использованием Docker. Docker позволяет запускать приложение в одинаковой среде независимо от операционной системы разработчика или сервера. В контейнеры могут быть вынесены PHP/Laravel-приложение, база данных PostgreSQL, frontend-сборка и сервер приложений. Это упрощает установку проекта, снижает количество ошибок, связанных с различиями окружения, и делает процесс запуска более воспроизводимым.

Для повышения производительности серверной части может использоваться RoadRunner в связке с Laravel Octane. RoadRunner является высокопроизводительным сервером приложений для PHP, написанным на Go. В отличие от классической модели, при которой PHP-приложение фактически инициализируется заново для каждого запроса, RoadRunner запускает пул долгоживущих worker-процессов. Laravel-приложение загружается один раз, остается в памяти и далее обслуживает множество запросов. Благодаря этому уменьшаются накладные расходы на повторную загрузку фреймворка, регистрацию сервисов, чтение конфигурации и подготовку окружения.

В рамках защиты проекта это можно объяснить следующим образом: применение RoadRunner ускоряет backend не за счет изменения бизнес-логики, а за счет изменения модели исполнения PHP-приложения. Запросы к API обрабатываются быстрее, поскольку готовые worker-процессы уже содержат загруженное приложение. Особенно заметным это становится для часто вызываемых маршрутов: публичной ленты, списка растений, задач ухода и административных списков. При этом для стабильной работы такого режима необходимо учитывать особенности долгоживущих процессов: не хранить пользовательские данные в глобальном состоянии, следить за памятью, корректно закрывать соединения и настраивать количество worker-процессов в соответствии с ресурсами сервера.

Для повышения надежности backend-части используются request-классы валидации, политики доступа и API resources. Request-классы проверяют корректность входных данных до выполнения основной логики. Policies определяют, имеет ли пользователь право выполнять действие над конкретным объектом. Resources формируют единый формат JSON-ответов для frontend-приложения. Такой подход делает серверную часть более структурированной, защищенной и удобной для сопровождения.

Качество backend-части контролируется с помощью PHPUnit, OpenAPI-контракта и Laravel Pint. PHPUnit используется для автоматизированной проверки API-сценариев. OpenAPI-спецификация описывает доступные маршруты и формат данных, что важно для согласованной работы frontend и backend. Laravel Pint применяется для поддержания единого стиля PHP-кода. В результате серверная часть проекта построена как полноценное REST API, обеспечивающее хранение данных, безопасность, бизнес-логику, производительный запуск через RoadRunner и интеграцию с клиентским приложением.

## Текст для защиты: frontend

Клиентская часть проекта реализована на Vue.js. Данный фреймворк был выбран, поскольку он позволяет создавать интерактивные одностраничные приложения с компонентной структурой. В проекте Vue.js используется для построения страниц ленты, личных растений, карточки растения, формы добавления и редактирования, задач ухода, профиля пользователя и административной панели. Компонентный подход позволяет разделить интерфейс на самостоятельные части и повторно использовать элементы, такие как карточки растений, кнопки, поля ввода, бейджи и виджеты.

Для сборки и запуска frontend-части используется Vite. Он обеспечивает быстрый запуск dev-сервера, удобную разработку Vue-компонентов и формирование production-сборки. Маршрутизация реализована через Vue Router. С его помощью определены основные страницы приложения и настроена проверка доступа к защищенным маршрутам. Если пользователь не авторизован, он перенаправляется на страницу профиля, а административный раздел доступен только пользователю с ролью администратора.

Состояние приложения хранится в Pinia. В проекте созданы отдельные хранилища для авторизации, растений, задач, социальной активности, календаря, статистики и административных данных. Такой подход позволяет отделить бизнес-состояние интерфейса от отдельных компонентов. Например, хранилище растений отвечает за загрузку ленты, фильтрацию, сортировку, создание, редактирование и удаление растений. Хранилище задач формирует мероприятия по уходу и отправляет запросы на создание записей в журнале ухода. Хранилище авторизации сохраняет токен и данные текущего пользователя.

Для взаимодействия с backend используется единый API-клиент. Он формирует HTTP-запросы, добавляет заголовки, передает Bearer-токен авторизованного пользователя, обрабатывает JSON-ответы и ошибки. Благодаря этому компоненты frontend не работают напрямую с низкоуровневой логикой запросов, а используют единый механизм обращения к API. Это упрощает сопровождение и делает поведение приложения более единообразным.

Для работы с формами применяются VeeValidate и Zod. Они используются для проверки вводимых данных на клиентской стороне, что помогает пользователю быстрее исправлять ошибки до отправки запроса на сервер. Для визуализации статистики используется Chart.js вместе с vue-chartjs. Иконки интерфейса реализованы через lucide-vue-next, а уведомления - через vue-sonner. Дополнительные библиотеки, такие как @vueuse/core, floating-vue, embla-carousel-vue и clsx, используются для адаптивности, всплывающих элементов, карусельных интерфейсов и удобной работы с CSS-классами.

Качество frontend-кода поддерживается с помощью ESLint и Prettier. ESLint проверяет JavaScript и Vue-компоненты на ошибки и нарушения правил, а Prettier обеспечивает единое форматирование. В результате клиентская часть проекта представляет собой современное SPA-приложение, которое обеспечивает удобный интерфейс для работы с растениями, задачами ухода, социальной лентой и административными функциями, а также взаимодействует с backend через REST API.
